## 1. 项目总览与现状分析

### 1.1 原始项目 OpenCode

开源 AI 编程助手，客户端/服务端架构，支持 CLI、Web、桌面三种模式。

**技术栈：** Bun 1.3+ / TypeScript / SolidJS / Tauri / SQLite + Drizzle ORM / Hono / Vercel AI SDK / Turborepo

**核心包结构：**

```
packages/
├── opencode/     # 核心：AI Agent、CLI、API 服务端、SQLite数据库
├── app/          # Web UI (SolidJS + Vite + Tailwind)
├── desktop/      # Tauri 桌面应用
├── ui/           # 共享 UI 组件库
├── sdk/js/       # JavaScript SDK
├── plugin/       # 插件系统
├── enterprise/   # 企业版 (SolidStart)
├── console/      # 控制台 (SolidStart + PlanetScale)
└── util/         # 通用工具
```

### 1.2 现状 vs 目标对比

| 维度       | 现状                | 目标                                           |
| ---------- | ------------------- | ---------------------------------------------- |
| 项目名     | OpenCode            | TpCode（研发版TpCode-U / 企业版TpCode-E）      |
| 用户体系   | 无用户概念          | 完整账号体系，与VHO统一                        |
| 认证机制   | 可选HTTP Basic Auth | JWT多用户认证                                  |
| 组织架构   | 无                  | 用户/部门/科室/角色/权限                       |
| 权限控制   | 仅AI工具权限        | RBAC + 数据隔离（我的/组/部门/全部）           |
| API Key    | 全局共享            | 每用户独立配置，后续集中控制                   |
| 项目管理   | 自动检测git仓库     | 解决方案→模块→页面 三级结构                    |
| 工作流     | 提示词→代码（断裂） | 提示词→计划→原型→审核→代码→测试→部署（端到端） |
| 审批流程   | 无                  | 一线→科室主任→信息科长→院长→研发审核           |
| 原型预览   | 无                  | 沙箱预览 + 版本回退                            |
| 文件浏览   | 右侧面板无限制      | 权限控制，默认隐藏                             |
| 知识库     | 无                  | 标准/竞品/领域知识/维基百科                    |
| 提示词安全 | 无                  | 禁用词过滤 + 塌缩防护                          |
| 数据采集   | 仅token统计         | 全量输入输出蒸馏记录                           |
| 监控       | 无                  | AI Agent状态大屏 + 使用效率统计                |

| 会话管理   | 无分类无排序        | 分类/分组/排序/权限隔离                        |

## 2. 改造愿景与目标

### 2.1 核心愿景

构建端到端全智能、全自动、全无人的 VibeCoding 平台：

```
提出需求(提示词) → 生成计划 → 生成原型 → 用户确认 → 审核审批 → 生成代码 → 测试验收 → 部署上线
```

**关键原则：**

- 端到端不是单纯代码范畴，是从需求到部署的完整链路
- 提示词基于使命/愿景/问题描述，不可直接描述具体功能（防止上下文塌缩）
- 分阶段打通：先打通「提示词→计划→原型」，再打通「测试→部署」
- 不能打通的环节，人工干预界限清晰

### 2.2 覆盖范围

| 覆盖对象     | 说明                                     |
| ------------ | ---------------------------------------- |
| 研发人员     | TpCode-U 研发版，日常编码                |
| 运维人员     | 服务器运维、部署管理                     |
| 项目人员     | 项目管理、进度追踪                       |
| 价值运营     | 方案评估、创收分析                       |
| 医院用户     | 门诊主任、财务科长等，提出需求、确认原型 |
| 全部内部人员 | 所有人使用TpCode                         |

### 2.3 第一期目标（下周内全员部署）

1. 完整账号体系（内部账号 + 医院账号 + 科室划分）
2. 与VHO统一账号体系对接
3. RBAC权限管理（用户/部门/角色）
4. 品牌重命名 → TpCode
5. 打通「提示词→计划→原型」链路
6. 沙箱预览 + 版本回退
7. 审核审批工作流
8. 用户级API Key配置
9. 文件浏览器权限控制/隐藏
10. 会话分类、分组、排序、权限隔离
11. 禁用词过滤与输入拦截
12. 蒸馏训练数据采集启动
13. 右侧文件浏览器去掉（权限控制）

### 2.4 第二期目标

1. 知识管理数据中心（标准/竞品/领域知识）
2. 端到端自动化流水线（测试→部署）
3. AI Agent监控大屏
4. 使用效率统计与质量评估
5. 日志采集与自动优化建议
6. 多模态语音合成辅助学习
7. 培训与考核模块
8. 自有编码大模型训练与部署

## 3. 总体架构设计

### 3.1 系统分层架构

```
┌──────────────────────────────────────────────────────────────────────┐
│                        前端展示层 (SolidJS)                          │
│  登录页 │ 解决方案选择 │ 模块/页面导航 │ 沙箱预览 │ AI对话          │
│  审批面板 │ 知识库 │ 监控大屏 │ 管理后台 │ 用户设置                 │
├──────────────────────────────────────────────────────────────────────┤
│                     API 网关层 (Hono + JWT)                          │
│  JWT认证 │ RBAC权限校验 │ 禁用词拦截 │ 蒸馏数据采集钩子            │
├──────────────────────────────────────────────────────────────────────┤
│                        业务逻辑层                                    │
│  用户服务 │ 组织架构服务 │ 解决方案服务 │ 审批工作流服务             │
│  知识库服务 │ 原型沙箱服务 │ 蒸馏采集服务 │ 监控统计服务             │
│  AI Agent服务(现有) │ Session服务(改造) │ 提示词安全服务            │
├──────────────────────────────────────────────────────────────────────┤
│                     数据访问层 (Drizzle ORM)                         │
│  用户/部门/角色表 │ 解决方案/模块/页面表 │ 审批表                   │
│  知识库表 │ 蒸馏数据表 │ 监控日志表 │ 禁用词表                     │
├──────────────────────────────────────────────────────────────────────┤
│                     存储层 (PostgreSQL)                               │
│  tpcode 主库（业务数据 + 蒸馏训练数据同库不同表）                    │
└──────────────────────────────────────────────────────────────────────┘
```

### 3.2 改造原则

1. **最小侵入**：不修改AI Agent核心逻辑（agent.ts、tool.ts、prompt/），在外围包装
2. **分层解耦**：新增功能以独立模块存在，通过中间件和服务层集成
3. **向后兼容**：保留CLI/TUI模式正常使用
4. **渐进打通**：先打通提示词→原型，再打通测试→部署，不能打通的人工干预
5. **数据主权**：所有输入输出全量记录，为自有编码大模型训练做准备

## 5. 账号体系与VHO统一对接

### 5.1 账号体系设计

| 账号类型     | 说明               | 示例                 |
| ------------ | ------------------ | -------------------- |
| 内部研发账号 | 研发/运维/项目人员 | zhangsan@tp.internal |
| 内部运营账号 | 价值运营人员       | lisi@tp.internal     |
| 医院管理账号 | 信息科长、院长     | admin@hospital-a     |
| 科室账号     | 门诊主任、财务科长 | menzhen@hospital-a   |

### 5.2 与VHO统一账号体系对接

**对接策略：**

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────┐
│   TpCode    │────→│  统一认证中心     │←────│    VHO      │
│  (JWT登录)  │     │  (SSO/OAuth2.0)  │     │  (JWT登录)  │
└─────────────┘     └──────────────────┘     └─────────────┘
                           │
                    ┌──────┴──────┐
                    │ 统一用户库   │
                    │ (PostgreSQL) │
                    └─────────────┘
```

**合并方案：**

1. **短期（下周）**：TpCode独立账号体系，预留VHO对接字段（`vho_user_id`）
2. **中期（2周内）**：实现SSO统一认证，TpCode和VHO共享用户库
3. **长期**：统一认证中心，所有系统通过OAuth2.0/OIDC接入

**对接字段预留：**

- `tp_user` 表增加 `vho_user_id` (text, nullable) 用于关联VHO用户
- `tp_user` 表增加 `external_source` (text, nullable) 标记用户来源（tpcode/vho/sso）

### 5.3 登录认证流程

```
用户输入账号密码 → TpCode后端验证 → 签发JWT → 前端存储Token
                                         ↓
                              后续请求携带Authorization头
                                         ↓
                              JWT中间件验证 → 提取user_id/role → 注入请求上下文
```

## 7. 端到端VibeCoding工作流

### 7.1 完整链路

```
┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
│提示词 │──→│ 计划 │──→│ 原型 │──→│ 确认 │──→│ 审批 │──→│ 代码 │──→│ 测试 │──→│ 部署 │
│(需求) │   │(方案)│   │(预览)│   │(用户)│   │(流程)│   │(生成)│   │(验收)│   │(上线)│
└──────┘   └──────┘   └──────┘   └──────┘   └──────┘   └──────┘   └──────┘   └──────┘
  第一期必须打通 ←──────────────────→          第一期打通 ←→   第二期 ←──────────────→
```

### 7.2 阶段一打通：提示词 → 计划 → 原型

**当前断裂点：** 提示词生成计划后，到原型代码这块断着，摩擦太重。

**打通方案：**

1. **提示词输入**：用户在页面上下文中输入需求描述
2. **AI生成计划**：Agent分析解决方案/模块/页面上下文，生成修改方案（伪代码+计划）
3. **原型生成**：Agent基于计划自动生成原型代码，部署到沙箱环境
4. **沙箱预览**：用户通过iframe直接查看原型效果
5. **版本回退**：原型不合适时，一键回退到上一版本

**沙箱机制：**

```
用户提交需求 → AI生成计划 → AI生成原型代码 → 部署到沙箱
                                                  ↓
                                          用户查看原型效果
                                           ↙          ↘
                                      满意：确认        不满意：回退
                                        ↓                  ↓
                                   提交审批           回到上一版本/重新生成
```

### 7.3 解决方案层级导航

用户登录后的操作流程：

```
登录 → 选择解决方案(项目) → 选择模块 → 选择页面
                                          ↓
                                   页面预览 + AI对话
                                          ↓
                                   输入修改需求
                                          ↓

                                          ↓
                                   确认效果 → 提交审批
```

## 9. 提示词安全与输入控制

### 9.1 禁用词过滤

**在输入框层面拦截，不合规提示词不进入模型。**

| 类别     | 禁用词示例                   | 拦截动作                       |
| -------- | ---------------------------- | ------------------------------ |
| 危险操作 | 删除、drop、truncate、rm -rf | 阻止提交，提示"该操作词被禁止" |
| 数据泄露 | 导出全部数据、dump database  | 阻止提交                       |
| 权限越界 | 修改管理员密码、提升权限     | 阻止提交                       |
| 自定义   | 管理员可配置的自定义禁用词   | 阻止提交                       |

**实现机制：**

```
用户输入提示词 → 前端禁用词检查 → 后端二次校验 → 通过 → 进入AI模型
                      ↓ 不通过                    ↓ 不通过
                 提示"该提示词不通过"          返回403拦截
```

### 9.2 上下文塌缩防护

**核心原则：** 提示词应基于使命/愿景/要解决的问题，不可直接描述具体功能。

| 类型     | 示例                                                     | 是否允许            |
| -------- | -------------------------------------------------------- | ------------------- |
| 愿景型   | "我们需要解决门诊排队等待时间过长的问题"                 | ✅ 推荐             |
| 问题型   | "患者反馈挂号后不知道什么时候轮到自己"                   | ✅ 推荐             |
| 功能型   | "给我设计一个排队叫号的页面，左边显示列表，右边显示详情" | ⚠️ 警告（可能塌缩） |
| 极具体型 | "在第三行第二列加一个按钮，颜色是#FF0000"                | ❌ 拦截             |

**防护策略：**

- 检测提示词中是否包含过于具体的UI描述（CSS属性、像素值、具体布局指令）
- 如果检测到，提示用户："建议从问题和目标出发描述需求，而非具体功能实现"
- 不强制拦截，但给出警告

### 9.3 特殊场景处理

| 场景                       | 处理方式                             |
| -------------------------- | ------------------------------------ |
| 标准化竞品方案导入         | 直接灌入，生成PPT/方案，允许具体内容 |
| 招投标文件                 | 允许具体内容，生成解决方案           |
| 极其抽象的愿景             | 允许，引导细化                       |
| 半吊子描述（半抽象半具体） | 警告，可能引起上下文塌缩             |
